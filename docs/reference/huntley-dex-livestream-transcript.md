## Key Concepts Discussed

### Context Windows as Arrays
- Context windows are literally arrays — ~200k tokens, ~16k overhead for system messages, ~16k for harness = **~176k usable**
- When you chat with the LLM you allocate to the array; when tools execute they auto-allocate
- Less sliding the window needs to do = better results
- There is a "dumb zone" — stay out of it by keeping one objective per context window

### Deliberate Memory Allocation (malloc analogy)
- First allocations should be deterministic: app specs → WORKLOG → relevant source
- Felt more deterministic when first allocations were made deliberately
- "Less is more" — agents.md should be ~60 lines

### Fresh Context vs Extended Context (Key Divergence)
- **Anthropic plugin approach:** Extends the same context window, uses compaction
- **Geoff's approach:** Fresh context per iteration, zero auto-compaction
- "One has zero auto-compaction ever. The other uses auto-compaction. Auto-compaction is lossy — it could remove the specs, remove the task and goal."
- Fresh contexts beat compacted contexts. Compaction is lossy.

### One Objective Per Context Window
- Not per iteration — per ENTIRE context window
- If you ask it to do too much, results in the working context degrade
- Want headroom for finalizing (git commands, pushes, test verification)
- "One context window, one activity, one goal"

### The Outer Loop / Orchestrator Pattern
- **Inner harness:** Claude Code (or any agent CLI) running the Ralph loop
- **Outer harness:** Orchestrator running Claude Code — a supervisor layer
- Opus is forgetful (e.g., always forgets translations in Loom)
- Solution: supervisor sees if the inner loop missed something → spawns another Ralph loop to fix it
- "The idea behind Ralph is an outer layer orchestrator, not just a loop"
- Loop could have branching: run main prompt → classify if X was done → jump to other prompts (add tests, fix tests, do translations)

### Human on the Loop (Not In the Loop)
- `dangerouslyAllowAll` = deliberately NOT injecting humans into the loop
- But run on isolated infrastructure (GCP VM with restricted blast radius)
- Watch like a fireplace — notice patterns, tune behaviors in/out
- "Never blame the model — always be curious about what's going on"
- Golden context windows — sometimes you get a perfect trajectory and want to save it

### Back Pressure & Test Runners
- Most test runners are trash — output too many tokens
- Only output failing test cases, truncate passing tests
- `head -N50` approach — don't tail, errors are often at the top
- Back pressure must be fast — wasted tokens = wasted time AND context

### Security (Lethal Trifecta)
- Network access + private data access + tool execution = danger
- Run on purpose-built GCP VMs with only deploy keys and API keys
- "Not IF it gets popped — WHEN. What's the blast radius?"
- No public IP, restricted permissions

### The Anthropic Plugin Architecture
- Uses a Stop hook that intercepts Claude's exit
- Injects the prompt again as a new user message → causes re-allocation
- Uses a `<promise>` completion signal
- Eventually hits compaction → model summary of "what it thinks is important" → keeps going
- Different behavior from the bash loop approach

### Loom (Geoff's Evolution)
- Remote ephemeral sandbox coding harness
- Self-hosted platform that creates its own remote agents ("weavers")
- Full SCM hosting + mirroring + agent harness + remote provisioner
- Source control as memory for agents
- "If you control the entire stack, you can modify it to your needs"

### T-Mux as Loopback
- LLMs know how to drive tmux — spawn sessions, split panes, scrape panes
- Use for automatic loopback: web server log + backend API log in splits
- Claude scrapes the panes for troubleshooting context
- Multiple Ralph loops in worktrees, one Claude scraping results and merging

## Architecture Diagram (Outer Loop)

```
┌─────────────────────────────────────────────┐
│              ORCHESTRATOR / SUPERVISOR        │
│  (team lead, bash script, or outer agent)     │
│                                               │
│  For each objective:                          │
│    1. Spawn Ralph loop (fresh context)        │
│    2. Watch for completion signal             │
│    3. Classify: did it do X? Y? Z?            │
│    4. If missed → spawn targeted follow-up    │
│    5. If quality check fails → reset + retry  │
│    6. If all green → merge, next objective    │
└─────────────┬───────────────────────┬─────────┘
              │                       │
    ┌─────────▼─────────┐   ┌────────▼──────────┐
    │   RALPH LOOP #1   │   │   RALPH LOOP #2   │
    │  (inner harness)  │   │  (follow-up loop)  │
    │                   │   │                    │
    │  allocate stack   │   │  "did you do       │
    │  implement        │   │   translations?"   │
    │  back pressure    │   │                    │
    │  fix / reset      │   │  targeted nudge    │
    │  complete         │   │  single objective  │
    └───────────────────┘   └────────────────────┘
```
